This is the PRD rewritten so Gemini 3 Pro (the autonomous agent) will execute the entire build and delivery pipeline end-to-end, with you as the human overseer for final approvals, risk checks and creative review. It keeps the previously approved X1-inspired luxury aesthetic and the ban on lending features. The agent responsibilities, prompts, tooling, QA checks, and rollback / escalation flows are explicit so the agent can run without human micro-management.
________________


1 — Mission & outcome (agent objective)
Agent mission: Convert Vaniland’s uploaded dataset into a live, performant, SEO-friendly, luxury real-estate portal with the “X1” visual language, using Next.js + Tailwind + Sanity (or equivalent), GSAP animations with reduced-motion fallbacks, and Algolia-style instant search. No lending features.
Primary outcomes the agent must deliver: deployable site (staging), seeded CMS content from the uploaded file, search index, CI artifacts (schema, migration scripts, Next.js code), automated tests, observability, and a human-review checklist for final signoff. Source data to ingest: uploaded dataset.
🏢 Vaniland Property Consultants
________________


2 — Agent capabilities required (what Gemini 3 Pro will do)
1. Parse and normalize the uploaded dataset, validate codes (e.g., 240181), infer missing fields, and produce seed JSON.
2. Create Sanity (or Strapi) schema files programmatically and push them to the target repo.
3. Generate a Next.js app scaffold (pages: /, /listings, /listing/[code], /contact, /admin) with Tailwind tokens matching the dark/gold palette.
4. Implement GSAP + ScrollTrigger animations with prefers-reduced-motion fallbacks and performance budgets.
5. Wire Algolia-like indexing (or provide a local fallback search index) and implement client search UI with URL-synced filters.
6. Build migration script that imports the uploaded file into CMS, logs dry-run output, and supports idempotent updates.
7. Run automated unit/integration tests, Lighthouse checks, and generate a deployment artifact to staging (Vercel or equivalent).
8. Create monitoring + alerting rules and produce a concise human review checklist and README.

6 — Data mapping & rules (agent must follow)
      * Canonical unique key: code (string). If duplicates in source: keep the most recently updated row (by file order) and flag duplicates in the report.
      * Price logic:
      * If status == for-rent, price = monthly rent (UGX).
      * If status == for-sale, price = sale price (UGX).
      * If sale_price missing, do not compute yield. Show “Contact for valuation” instead.
      * Field normalization:
      * bedrooms, bathrooms → integers (if parse fails, set null and flag).
      * location.neighborhood and district normalized using a canonical list (agent will attempt fuzzy-match against an internal Kampala districts list; unmatched entries are flagged).
      * Images: If no image provided, the agent must mark images: [] and create a task entry for human to supply local photography. Agent may temporarily use a neutral architectural placeholder with metadata tag "placeholder": true.

7 — CMS schema the agent must produce
property fields (required): code, title, type, status, price, currency, bedrooms, bathrooms, location{district,neighborhood,lat,lon}, images[], description, amenities[], featured, premium, condition, meta, published, created_by, updated_at.
lead fields: id, created_at, name, email, phone, propertyCode, goal, budget, utm, consent.
Agent must also include validation rules (e.g., code unique, price >= 0, status enum). Save schema in repo at /cms/schema/.
________________


8 — Code generation rules for the agent
      * Next.js pages must be SSR or SSG with ISR (revalidate every 10 minutes by default).
      * Use next/image for images. Configure sizes: [320,512,768,1024,1600]. Convert to WebP.
      * Tailwind tokens:
      * --color-deep-navy: #0B1222
      * --color-gold: #C1A36F
      * --color-slate (choose neutral slate)
      * All headings must be server-rendered text; animations apply to already-rendered nodes.
      * GSAP usage must include checks for prefers-reduced-motion and a data-low-power toggle for mobile devices.
      * Code must include unit tests for price parsing, yield calculation (only when both sale and rent present), getStaticProps correctness for /listing/[code], and an integration test verifying POST /api/leads persists to CMS.
________________


9 — Search & filter behavior (agent to implement)
      * Debounce: 300ms.
      * Filters: status, type, district, priceMin, priceMax, bedrooms.
      * Sorting: relevance, price_asc, price_desc, newest.
      * URL format: /listings?status=for-rent&district=Kira&beds=2&page=1.
      * For Algolia alternative, produce search-index.json and an endpoint /api/search that performs filtering server-side for staging if Algolia keys are not provided.

10 — Automation, CI & deployment (agent responsibilities)
      * Every change = commit on branch ai/vaniland-build.
      * CI pipeline:
      * Install, lint, run unit tests
      * Build Next.js
      * Run Lighthouse (Performance & Accessibility)
      * Deploy to staging on success
      * Agent must enforce pre-merge rules: tests pass, Lighthouse accessibility ≥ 85, no console errors in E2E smoke tests.
      * If Lighthouse perf < 50, agent auto-runs optimization: image resizing, remove non-critical animations, enable script-splitting.

11 — Monitoring, SLOs & alerts (agent must configure)
      * Uptime: staging URL check every 5 minutes; alert on 3 consecutive failures.
      * Form health: verify /api/leads returns 200 and pushes a test lead daily; failure alerts.
      * Error budget: send alert if error rate > 1% over 10 minutes.
      * Reporting: agent generates weekly status report and attaches to the repo as /reports/status-YYYY-MM-DD.md (while in active dev).

12 — Security & privacy rules (non-negotiable)

14 — Failure modes & escalation (what the agent must do)
         * If migration encounters >5% malformed rows → abort and produce CSV of malformed entries for human review.
         * If deploy fails or tests fail → revert to previous passing build and create a PR with failing logs.
         * If Algolia keys missing → create local index and flag for human to approve Algolia later.
         * Critical incidents: agent creates an incident report and notifies the human overseer by email/Slack webhook with triage steps.
